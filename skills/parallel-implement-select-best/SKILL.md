# parallel-implement-select-best

新規フェーズの実装で、最初から 1 案に賭けず、複数案を並列実装させてから最良案を採用するためのスキル。

## 使いどころ

- `Phase 1.5` 以降の新しい実装に着手するとき
- 仕様は明確だが、実装の切り方に複数の有力案があるとき
- 初手から review-fix loop に入る前に、実装案の質を競わせたいとき

既存実装の指摘修正にはこのスキルを使わず、`skills/review-fix-loop/SKILL.md` を使う。

## 原則

- 並列数の初期値は **3 体**
- 全員に同じ完了条件を渡す
- ただし ownership は完全に分離し、各案は独立 worktree で作る
- 採用するのは **1 案だけ**
- 不採用案は統合しない
- 採用後に終わりではなく、その案を正本として `review-fix-loop` に接続する

## 1. 事前整理

実装を競わせる前に、メインが以下を確定する。

1. 今回の対象フェーズと完了条件
2. 設計 docs と README 上の制約
3. 変更対象ファイル群
4. 絶対に守る invariant
5. 採点観点

採点観点は実装前に固定する。

### 推奨採点観点

- docs / contract との整合性
- 正確性
- 既存コードとの一貫性
- 保守性
- 変更範囲の適切さ
- テストの質
- 後続 review-fix loop に載せやすい明瞭さ

## 2. Candidate Implement Round

3 体前後の worker を並列で起動する。

- 各 worker は別 worktree
- 全員に同じ仕様を渡す
- ただし「どう実装するか」は各 worker に任せる

worker には次を明示する。

- 今回の仕様
- 完了条件
- 変更してよい範囲
- 必須テスト
- 他 worker の変更を気にせず、自分の案を完結させること

## 3. Candidate Review

3 案が上がったら、メインが各案を比較する。

確認項目:

- docs 違反がないか
- テストが通るか
- 差分が過剰でないか
- 既存パターンに素直に乗っているか
- 危険な hidden assumption がないか

必要ならここで軽いレビューを追加してもよいが、主目的は **採用候補を 1 つに絞ること**。

## 4. Scoring

3 案を同じ観点で採点し、1 位を選ぶ。

### 最低限の出力

| candidate | score | summary |
|---|---:|---|
| wt-a | 8.7 | 強み / 弱み |
| wt-b | 7.9 | 強み / 弱み |
| wt-c | 8.2 | 強み / 弱み |

採点時の注意:

- 「動いた」だけで選ばない
- 変更量が少ないだけでも選ばない
- docs 違反や contract 破壊がある案は失格にしてよい
- 1 位に明確な欠点が残るなら、採用前にメインが補正する前提で選んでよい

## 5. Winner Selection

採用手順:

1. 1 位の案を採用候補にする
2. 採用候補をメインが再確認する
3. 必要なら採用前に小さな補正を入れる
4. それを正本として統合する

不採用案は merge しない。
削除や cleanup は repo ルールとユーザー方針に従う。

## 6. Hand-off to Review Loop

採用案を統合したら、その時点で終了しない。

次のステップは必ず `review-fix-loop`:

1. 採用案を正本にする
2. 6 体前後でレビューを回す
3. findings を精査する
4. 必要なら 3〜4 分割で修正する
5. 再統合・再レビューを回す
6. `High` が 0 になるまで続ける

## 7. 典型的な使い分け

- **既存実装の修正 round**:
  - `review-fix-loop`
- **新しい phase の最初の実装 round**:
  - `parallel-implement-select-best`
  - その後 `review-fix-loop`

## 禁止事項

- 3 案を混ぜて「良いとこ取り merge」をしない
- 採用前に採点基準を変えない
- 不採用案まで全部統合しない
- 採用案の統合後 review-fix loop を省略しない
