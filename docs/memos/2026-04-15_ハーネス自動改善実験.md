## スクリプトのバグと修正

### バグ1: done ファイル残留

**症状**: PR84 再実行時に前回の `done` ファイルが `/tmp/eval-pr84-*-logs/` に残っており、wait loop が即座に「全完了」と誤判定。

**原因**: `harness-run.sh` の Step 1 で `mkdir -p "$LOG_DIR"` はするが、前回の `done` / `exit_code` を削除していなかった。

**修正**: `rm -f "$LOG_DIR/done" "$LOG_DIR/exit_code" "$LOG_DIR/session.jsonl"` をStep 1に追加。

### バグ2: `&` と `run_in_background` の二重使い

**症状**: `bash script.sh &` + Bash ツールの `run_in_background: true` を併用すると、親シェル終了時にフォークしたスクリプトがSIGHUPで殺される。

**原因**: `&` でフォークした子プロセスが、Bash ツールのプロセス管理により親と一緒に終了させられる。

**修正**: `&` を外し、`run_in_background: true` のみで実行。PR84とPR59を順次実行に変更。

### バグ3: diff.patch が不完全

**症状**: 複数コミットのエージェント（auto-commit + 実装コミット）で、`diff HEAD~1` が最後のコミット分しか取れない。

**修正**: base commit からの full diff (`git diff <base>`) に差し替え。

## 自動改善パイプラインの設計

### 2段階の課題

**Phase 1: 改善要素の抽出** — 何を直すべきか正しく特定する

**Phase 2: ハーネスへの反映** — 全て ai-rules に書く（rules 固定）

### 問題検出の4基準

diff 比較だけでなく、複数の基準で問題を検出する:

1. **実PRとの差異**: agent diff vs 実PR diff
2. **ルール違反**: agent output vs frontend-rules/checklist
3. **要件未達**: agent output vs requirements.md/タスク指示文
4. **設計方針違反**: agent output vs architecture-guide.md

### 原因特定の決定木

問題ごとに以下の順で原因を絞り込む:

```
問題を1つ選ぶ
  │
  ├─ 関連情報が既存 docs/rules に書かれているか？
  │    NO → NO_RULE
  │    YES ↓
  │
  ├─ エージェントはそのファイルを読んだか？（file-access.log）
  │    NO → NOT_READ
  │    YES ↓
  │
  └─ ミスした時エージェントは何を考えていたか？（session.jsonl thinking）
       ├─ 矛盾した推論 → MISUNDERSTAND
       ├─ 考慮せず実装 → FORGOT
       ├─ rate limit / hook block / token limit → SYSTEM_BLOCK
       └─ 上記以外 → OTHER（理由必須記載）
```

### Finding 形式（中間出力）

```markdown
## Finding: <短いタイトル>
- Category: GEN / MIG / API / TEST / SEO
- Severity: HIGH / MEDIUM / LOW
- Problem: <何が悪かったか>
- Evidence:
  - docs: <確認したドキュメントと該当有無>
  - file-access: <該当ファイルの読み込み有無>
  - thinking: <該当 thinking の要約>
- Root Cause: NO_RULE / NOT_READ / MISUNDERSTAND / FORGOT / SYSTEM_BLOCK / OTHER
- Detail: <原因の詳細>
```

### ai-rules 形式（最終出力）

`docs/frontend-rules/` と同一形式。`generate-checklist.sh` でチェックリストに自動変換される。

```markdown
## AI-XXX-NNN: <タイトル>
- Level: MUST / SHOULD
- Intent: ルールの意図
- Rule: チェックリスト項目になる1行の説明
- Why: <Finding への参照と根拠>
- Detection: <違反の検知方法>
- Source: <実験ID>
```

ID体系: `AI-GEN`, `AI-MIG`, `AI-API`, `AI-TEST`, `AI-SEO`

## 分析プロンプト比較実験

### 実験変数

- **固定**: ai-rules 形式、全13体に適用、findings テンプレート
- **変数**: サブエージェントに渡すプロンプトの指示の具体性

### 3プロンプト

| ID | 方向性 | 指示の具体性 |
|---|---|---|
| A | 3段階分析（問題検出→原因特定→ルール化） | 中（Step分離 + 4基準列挙） |
| B | 一括分析（簡潔指示） | 最小限（データ渡して一発） |
| C | 決定木分析（段階的原因特定） | 最大（決定木 + データソース明示） |

共通入力データ:
- actual_pr_diff — 実PRのdiff
- agent_diff — エージェントのdiff
- session_jsonl — セッションログ
- file_access_log — ファイルアクセス記録
- hook_blocks_log — hookブロック記録
- rules_checklist — ルール/チェックリスト
- requirements — 要件定義
- architecture_guide — アーキテクチャガイド
- findings_file — 出力先テンプレート

ファイル: `docs/harness-eval/analysis-prompts/prompt-{a,b,c}.md`

### 実行方法

各プロンプト × 13体 = 13回の `claude -p` 実行。キュー方式で並列数を制御。

```bash
MAIN_SURFACE=surface:46 bash scripts/analysis-run.sh a 3
MAIN_SURFACE=surface:46 bash scripts/analysis-run.sh b 3
MAIN_SURFACE=surface:46 bash scripts/analysis-run.sh c 3
```

### 評価用ブランチ

```
feature/nishimoto-dev-tools     ← base（インフラ全部入り）
  ├── analysis-prompt-a         ← A の ai-rules が入る
  ├── analysis-prompt-b         ← B の ai-rules が入る
  └── analysis-prompt-c         ← C の ai-rules が入る
```

分析後の流れ:
1. 各プロンプトの findings を集約 → ai-rules 生成
2. 対応するブランチに ai-rules をコミット
3. 各ブランチから harness-run.sh で eval worktree 作成 → 実装エージェント実行
4. 実装結果を採点・比較 → 最も効果的なプロンプトを特定
