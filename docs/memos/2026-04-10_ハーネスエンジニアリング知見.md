# ハーネスエンジニアリング知見

## 調査日
2026-04-10

## ハーネスエンジニアリングとは

**Agent = Model + Harness**

AIモデルを取り巻く「モデル以外のすべて」を設計する技術。Mitchell Hashimoto（HashiCorp創業者）が2026年2月に命名。

- 同じモデルでもハーネスを変えるとSWE-benchスコアが **22点変動**、モデルを変えても **1点程度**
- モデル性能よりハーネス設計の方が成果に大きく影響する

## プロンプトエンジニアリングとの違い

| 世代 | 概念 | 比喩 |
|---|---|---|
| 第1世代 | プロンプトエンジニアリング | 馬への命令 |
| 第2世代 | コンテキストエンジニアリング | 地図・道路標識 |
| 第3世代 | ハーネスエンジニアリング | 手綱・鞍・柵・道路そのもの |

最大の違いは **強制力**。CLAUDE.mdで「anyを使うな」→ 忘れる。linterでブロック → 物理的に使えない。

## フレームワーク

### フィードフォワード（事前予防）
- **CLAUDE.md** — 行動制約、規約
- **Skills** — 必要な時だけ情報を開示（段階的知識開示）
- **settings.json** — hooks、デフォルト設定

### フィードバック（事後検証）
- **Hooks** — pre-commit時のlint、セキュリティスキャン
- **テスト・型チェック** — バックプレッシャー機構
- **サブエージェント** — コンテキストファイアウォール
- **MCP** — 外部ツール接続（必要最小限に）

## lintで縛りすぎることの弊害

### 無限ループ
- lintエラー → 修正 → 別のlintエラー → 修正 → 元に戻る、の繰り返し
- コンテキストウィンドウを焼き尽くす

### コンテキスト汚染
- AGENTS.mdの約74%がエージェントの時間を無駄にしている（ctxlint調査）
- linterが既にカバーしているルールをCLAUDE.mdにも書く → 二重管理でノイズ
- コンテキスト長が70%を超えると出力品質が低下し始める

### 過度に従順になる
- ETH Zurich研究: AGENTS.mdを使うとタスク成功率が約3%低下、推論コスト20%以上増加
- 不要な指示に従うことで本来の問題解決が疎かに

## 適切な制約のバランス

### 核心原則
> **不変条件を強制し、実装を細かく管理しない**

### やるべきこと / やるべきでないこと

| やるべき | やるべきでない |
|---|---|
| アーキテクチャの不変条件をHooksで強制 | 全スタイルルールをCLAUDE.mdに羅列 |
| エラーメッセージに修正方法を含める | 「lint failed」だけの曖昧なエラー |
| 成功時はサイレント（出力なし） | 成功時に冗長なログ出力 |
| 連続リトライ回数を制限 | 無制限リトライ |
| linterがカバーするルールはHooksに任せる | linterとCLAUDE.mdに同じこと二重記述 |

### エラーメッセージ = プロンプト
エラー出力は「ルール → 違反箇所 → 修正方法」の三点セットで設計する。曖昧なエラーはエージェントの無駄な試行錯誤の原因。

### Hooks の制御フロー

| Exit Code | 動作 | Claudeへの影響 |
|---|---|---|
| 0 | 成功、続行 | サイレント（コンテキスト消費なし） |
| 2 | ブロック | stderrがClaudeにフィードバック |
| その他 | 続行 | 非致命的警告 |

## 有識者の見解

- **Mitchell Hashimoto**: 「高レベルの制約を与え、低レベルの実装は任せる」
- **Martin Fowler**: 「制約が解決空間を狭めることでエージェントは逆に生産的になる」ただし過剰は逆効果
- **OpenAI**: 「ガードレールは現在のモデルの限界に対するヒューリスティック。モデル改善で不要になる可能性がある」
- **Anthropic**: CLAUDE.md = アドバイザリー（助言的）、Hooks = 決定論的保証（必ず実行）

## ベストプラクティス

1. **出荷偏重で始める** — 完璧設計より実装しながら失敗に対応
2. **成功時は静かに、失敗時のみエラー** — コンテキスト圧迫を防ぐ
3. **計算的制御を優先** — linter/テスト等の決定的制御を先に
4. **コンテキスト管理** — 長時間タスクでの窓枯渇を防ぐ
5. **モデル進化に追従** — 不要な制約は除去すべき
