## 重要な発見

### 1. rulesのpathsは注入されたが無視された

設定Cのセッションログを分析した結果:
- `.tsx` ファイルをReadした時点でrules（CSS責務、コンポーネント設計、ページ実装、移植ルール）が**全て注入されていた**
- しかしエージェントの発言に**ルールへの言及が一切ない**
- 「十分な情報が揃いました。実装します」で即座に書き始めた

### 2. CLAUDE.mdに直接書いてある方が従いやすい

設定Aのエージェントは:
- 「next-intlの辞書に入れます」→ 多言語ルールに従っている
- 「CSS Modulesを使うパターンがある」→ 既存パターンを分析
- 「`<picture>` でSP/PC出し分け」→ Tailwind/Next.jsルールに従っている

**CLAUDE.mdに最初から書いてあると「意識に入る」。rulesで途中注入されると「読んだファイルの付属情報」として扱われ、実装方針に反映されにくい。**

### 3. 移植タスクとrulesのpathsの相性問題

実行順序の分析:
```
1. Read: about.vue       ← .vueなのでどのrulesもトリガーされない
2-7. プロジェクト構造調査
8. Read: page.tsx         ← ここでrulesが注入される
...
26. Write: page.tsx       ← 実装完了
```

元ファイル（.vue）を読む時点ではrulesが届かない。既存の.tsxファイルを読んだ時点で注入されるが、その頃にはエージェントが方針を決めてしまっている。

### 4. CLAUDE.mdの情報量を削ると品質が下がる

- 設定A（154行）: スコア3.3
- 設定C（27行）: スコア2.7
- 0.6ポイントの差。rulesで補おうとしたが補えなかった

### 5. チェックリスト方式はeffort mediumでは機能しない

設定Bのチェックリストを実際にコピーして使った形跡なし。能動的にチェックリストを使うスキルが必要だが、effort mediumではそこまでの手順を踏まない。

## 結論

1. **CLAUDE.mdは削らない方がいい**。154行は多く見えるが、エージェントの品質に直結
2. **rulesのpaths注入は、CLAUDE.md常時ロードの代替にはならない**。注入されても無視される
3. **rulesのpathsは「補助情報」として使うべき**。メインのルールはCLAUDE.mdに残す
4. **移植タスクではpathsのトリガーが遅れる**。元ファイルの拡張子が異なるため
5. **effort mediumではタスク指示の具体性がハーネス設定より影響が大きい**

## 今後の改善案

- CLAUDE.mdは現在の154行を維持しつつ、rulesを**補助的に**追加
- pathsなし（グローバル）のruleで最重要ルールを常時注入
- effort highでの再実験
- タスク指示に「コンポーネント設計方針に従って構造を検討してから実装」を含める
