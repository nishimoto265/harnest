# ルールライフサイクル設計

## 背景

ハーネスのルール（ai-rules, ESLint, hooks）を継続的に管理する仕組みが必要。
ルールは追加するより削除する方が難しく、放置すると膨張してチェックリスト疲労を引き起こす。

## 参考: 現実世界のルール管理

### 各分野の共通パターン

| 観点 | 共通パターン |
|---|---|
| 追加 | 根本原因分析 or エビデンスに基づく。感覚的な追加を避ける |
| 削除 | 定量データで有効性確認。形骸化（遵守率低下/効果なし）を検出 |
| 統合 | ルールを「仕組み」に昇華（ポカヨケ, lint, 自動化） |
| 有効性 | 結果指標（不良率）+ プロセス指標（遵守率）の両面で測定 |
| ライフサイクル | 追加→有効期間→レビュー→更新/廃止の明示的サイクル |

### トヨタ (TPS)
- 削除: 作業者が「このステップは価値を生んでいない」と提案 → 省いても品質・安全に影響ないか実験的に確認
- 統合: 複数チェック項目をポカヨケ（物理的防止機構）に置き換え → ルール自体を不要にする
- 測定: 不良率・ライン停止回数で直接測定

### WHO 手術チェックリスト (Gawande)
- 追加のキラー質問: 「この項目がなければ致命的なエラーが起きるか」
- 削除基準: 全員が内面化し、チェックなしでも100%遵守される項目は削除候補
- 核心: 「チェックリストは短くなければ使われない」

### 航空 (FAA)
- チェックリスト長の限界: ワーキングメモリの限界（5〜9項目）以内。超える場合はフェーズ分割
- 削除: 航空機の設計変更や自動化で不要になった項目を削除

### 法律 (サンセット条項)
- ルールに有効期限を組み込む。効果が実証されなければ自動失効

## 参考: Googleのコード品質管理

### 5層の仕組み

```
① 自動フォーマッタ (gofmt, google-java-format)
   → 議論の余地なし。保存/コミット時に自動適用
② 静的解析 (Tricorder, Error Prone)
   → CIでチェック。レビュー画面にインラインコメントで表示
③ Readability認定
   → 言語ごとに人間が認定。未取得者のコードは認定者の承認が必須
④ コードレビュー (Critique)
   → Owner承認 + 技術LGTM + Readability LGTMの3つが必要
⑤ 文化
   → 「コードは個人ではなく組織のもの」
```

### Tricorder の false positive 管理

- レビュー画面に「Not useful」ボタンがある
- **10%以上が Not useful と評価された解析器は無効化される**
- これが「ルール精度が高いものだけ残る」仕組みの実装
- 偽陽性率が実質ゼロのもののみ Error（ビルドブロック）。それ以外は Warning（表示のみ）
- 原則: 「開発者の信頼を失わないこと」が最優先

### Error / Warning の判断基準

| レベル | 対象 | 偽陽性率 |
|---|---|---|
| Error（CIブロック） | バグの原因、セキュリティリスク | 実質ゼロ |
| Warning（表示のみ） | 品質改善だが文脈依存 | 低〜中 |
| Off | 好みレベル or フォーマッタが処理 | - |

### 我々のハーネスとの対応

| Google | 我々のハーネス | 備考 |
|---|---|---|
| 自動フォーマッタ | Prettier (既存) | 対応済み |
| Tricorder/Error Prone | ESLint (error/warning) | 一部対応 |
| スタイルガイド | ai-rules (docs) | 対応済み |
| Readability認定 | なし | AIに認定制度は不可 |
| Critique + 3承認 | hooks (PreToolUse) | 一部対応 |
| Rosie (一括修正) | なし | 未対応 |
| Not useful ボタン | compliance-check | **これが欠けている → 設計する** |

## lint/テストの厳格度について

### なぜ人間には厳格にしなかったのか

| 理由 | 説明 |
|---|---|
| 認知コスト | lint修正でフロー状態が中断。復帰に15-25分（最大の理由） |
| 整備コスト | Google規模の専任チーム + インフラ投資なしには維持困難 |
| 価値の不明瞭さ | スタイル系ルールの厳格化が品質にどの程度寄与するかのエビデンスが弱い |
| 暗黙知で補完 | 経験ある開発者は「危険なパターン」を直感で避けられる |

### AIだと事情が変わる部分

| 観点 | 人間 | AI |
|---|---|---|
| ルール修正の認知コスト | 高い（フロー中断） | ゼロ |
| false positiveへの反応 | フラストレーション → fatigue | なし |
| 暗黙知 | 経験で補完できる | 明文化が必要 |
| 一貫性 | 疲労で低下するが基本安定 | セッション間で不安定 |

### ただしAIでも厳格すぎると害がある

| 現象 | 事例 |
|---|---|
| false positiveによる修正ループ | react/no-danger で既存ファイル誤検知 → 4-6回の無駄な修正 |
| ルール過多でタスク未着手 | 方式C（101項目）→ 3体中2体が実装に到達できず |
| eslint-disable乱用 | エラー解消の最短経路としてdisableを挿入 |

### 結論

- 人間にもAIにも共通: **量を絞って精度の高いルールだけを厳格にする**
- AIの方が閾値を高くできる（認知コストゼロ）が、100項目は多すぎる
- **ルールの精度（precision）が高いものだけを error にする** — 人間・AI共通の原則

## ルールライフサイクルの設計

### 段階的採用フロー

```
ルール作成（方式Bバッチで自動生成）
  ↓
【お試し段階】(warning相当)
  - チェックリストの「参考ルール」セクションに配置
  - AIに「該当する場合のみ考慮」として提示
  - PR後に compliance-check で遵守/違反/該当なし を自動記録
  ↓
該当PR 10件で判定:
  - 正解率 90%以上 → 本採用
  - 不適切が2件連続 → 即除外
  - 30PR経過しても該当0件 → 具体的すぎる → 見直し or 削除
  ↓
【本採用】(error相当)
  - チェックリストの「必須確認」セクションに移動
  - lint化可能なものはESLintに移行
  ↓
四半期レビュー:
  - 遵守率100%が3ヶ月続く → AIが内面化済み → 削除候補
  - 遵守率が急落 → モデル更新の影響 → 要調査
  - false positive率10%超 → 除外 or lint条件修正
  ↓
【削除/降格】
  - ソフトデリート（非アクティブ化）→ 一定期間後に完全削除
  - 削除後にインシデント発生 → 復元
```

### 遵守率の自動計測（人間判断なし）

```
PR完了
  ↓
Phase 1: grep/lint（確実）
  - Detection にgrepコマンドがあるルール → 自動実行
  - 結果を compliance.jsonl に記録
  ↓
Phase 2: 軽量AI判定（grep不可のルール）
  - claude -p (haiku, low effort) でdiff + ルールを読み遵守判定
  - compliant / violated / not_applicable をJSON出力
  - compliance.jsonl に追記
  ↓
蓄積データから自動集計
  - 昇格条件を満たしたら自動フラグON
  - 降格条件を満たしたら自動フラグON
```

### チェックリストの構成

```markdown
# 必須確認（本採用ルール）
- [ ] AI-GEN-005: YAGNI（元コードにない機能追加禁止）
- [ ] AI-GEN-013: getTranslations集約
...

# 参考ルール（お試し — 該当する場合のみ考慮）
- AI-GEN-019: web2ソースリストアップ [正解率: 7/8 = 87.5%]
- AI-API-007: 型定義ドメイン分類 [正解率: 3/3 = 100%]
...
```

## 次のステップ

1. compliance-check.sh のプロトタイプ作成（grep + AI判定）
2. 既存13体の実験データで精度検証（人間採点との一致率）
3. ai-rules の各ルールにメタデータ追加（追加日, Source-count, サンセット日）
4. チェックリストを「本採用/お試し」に分離する仕組みの実装
