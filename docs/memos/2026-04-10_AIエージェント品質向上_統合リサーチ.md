## 1. マルチエージェントオーケストレーション

### Multi-Agent Debate (MAD)
- ICLR 2025の分析: MADは**単純なSelf-Consistency（多数決）を安定的には超えられない**
- ただし**判定型タスク**（安全性判定、応答評価等）ではエージェント数を増やすほど精度向上
- DMAD（多様性導入）が従来MADより少ないラウンドで高精度を達成

### Self-Refine / Reflexion
- Reflexion: HumanEvalのpass@1で**GPT-4ベースラインから約91%まで向上**
- Multi-Agent Reflexion (MAR): HumanEval pass@1 **76.4 → 82.6 (+6.2pt)**、ただしAPIコスト約3倍
- MARS: MADと同等精度で**トークン使用量・推論時間を約50%削減**

### 役割ベース協調の主流化
- Gartner 2025: 企業AIプロジェクトの**72%がマルチエージェント採用**（2024年は23%）
- PwC + CrewAI: コード生成精度が**10% → 70%（7倍改善）**
- MetaGPT: HumanEval Pass@1で**85.9-87.7%**

## 2. SE手法のAIエージェント適用

### TDD（テスト駆動開発）— 最も効果的

- TDAD（SWE-bench Verified）: **リグレッション率70%削減**（6.08% → 1.82%）

- obra/superpowers: GitHub **121,000+ Stars**

### Property-Based Testing — TDDの補完
- PGS: pass@1で**23.1%~37.3%の相対改善**
- PBT + EBT併用: バグ検出率**81.25%**（各単体68.75%）

### Mutation Testing — テストの質を検証
- Meta社: 10,795クラスからミューテーション生成、エンジニアの**73%がテストを承認**
- カバレッジ100%でもミューテーションスコア**わずか4%**のケースあり
- Cursorにミュータント情報を見せると**70% → 78%に改善**

### コードレビュー自動化
- HubSpot Sidekick: フィードバック時間**約90%短縮**、エンジニア**80%が承認**
- AI生成コードは人間より**1.7倍多い問題**（CodeRabbit分析）
- AIレビューツール: 実行時バグの**42-48%を検出**

### AI生成コードの品質問題
- コード重複が**8倍に増加**（GitClear 2億行分析）
- リファクタリング割合が**25% → 10%未満**に低下
- ガードレールなしだと**6ヶ月でバグ密度35-40%増加**

### SRE / エラーバジェット
- Microsoft Agent Governance Toolkit: 安全性SLI **99%**下回ると自動制限
- Azure SRE Agent: 自動根本原因分析とインシデント対応をGA提供
- Agent SRE成熟度モデル: Read-Only → Advised → Approved → Autonomous の4段階

### 半形式的検証（Meta）

- RubberDuckBench: **87%**（+9~11ポイント改善）

## 3. 異分野からの着想

### 航空業界: CRM（クルー・リソース・マネジメント）
- **権威勾配の排除**: レビューエージェントが生成エージェントの権威に遠慮せず異議を唱える構造
- **ブラインドレビュー**: 生成元が誰かを隠してコード品質のみで判断
- **OODAループ**: Observe → Orient → Decide → Act をエージェント動作サイクルに構造化
- Bruce Schneier指摘: AIエージェントはOrient段階（文脈理解）が最も弱い

### 医療業界: ダブルチェック・チェックリスト
- WHO手術安全チェックリスト: **死亡率47%削減**
- **Independent Double Check**: 二人が独立に確認し結果を照合（確証バイアス排除）
- **ACH（競合仮説分析）**: バグ修正時に複数の原因仮説を系統的に排除
- **スイスチーズモデル**: 複数の独立した防御層でバグを多段階で捕捉

### 製造業: トヨタ生産方式
- **自働化（Jidoka）**: 異常検知で即停止、根本原因修正後に再開
- **ポカヨケ**: 構造的にミスを不可能にする（型システム、テンプレート、禁止パターン）
- **アンドン**: エージェント動作状態のリアルタイム可視化、異常時に人間に通知
- **なぜなぜ分析**: バグ修正で5 Whysによる根本原因追求
- Woven by Toyota「MISRA Copilot」: MISRA違反の**約80%を自動修正**

### 軍事: ミッションコマンド・構造化分析
- **Commander's Intent**: 「何を達成するか」を伝え、「どう達成するか」は現場に委ねる
- **Staff Estimate**: 複数のCOA（行動方針）を評価してから最適案を選択
- **Devil's Advocacy**: 意図的に問題点を探す敵対的レビュー

### 宇宙・原子力: 多重冗長性
- **NASA TMR（三重モジュラー冗長）**: 3つの独立システムで同一処理、多数決で出力決定
- DeepMind AZdb: **行動多様性**を持つエージェント群でElo +50、パズル解決率2倍
- **段階的権限モデル**: リスクレベルに応じて自動実行/確認実行/人間承認を使い分け

### チェスエンジン: 相補的アンサンブル
- Stockfish（計算的・精密）× Leela（戦略的・直感的）の組み合わせが最善
- AIコーディングへの応用: 型安全性重視エージェント × 設計パターン重視エージェント × パフォーマンス重視エージェントの合議

### 科学: ピアレビュー
- **Blind Review**: 実装者の「思考過程」を隠してコードのみで評価
- **Pre-registration（事前宣言）**: コードを書く前に期待動作を宣言（TDDと同根）
- **再現性**: 別エージェントが仕様からテストを書いて通るか検証

## 6. 自己改善エージェント

### メモリの3層設計
- **Episodic memory**: 過去試行の軌跡、失敗理由、成功条件
- **Semantic memory**: 複数の失敗から抽象化した教訓
- **Procedural memory**: 実行可能スキルやAPIとして保存

### 実装パターン
```
trial → feedback → reflection → memory update → retry
```

### 最新研究
- **ContractSkill**: スキルを事前条件・事後条件・回復規則つき契約として表現
- **MemSkill**: 記憶の抽出・要約・統合・削除を「学習可能なスキル」として扱う
- **EvoSkills / SkillFoundry**: スキルをmulti-file artifactとして生成し、verifierと共進化

### 核心的洞察
> 同じ失敗を防ぐには「全文ログ保存」では足りず、**failure abstraction**（失敗の抽象化）、**retrieval discipline**（検索規律）、**skill packaging**（スキルのパッケージ化）が必要

## 7. 人間チームパターンのAI適用

### Conway's Law → Inverse Conway Maneuver
- エージェント編成を先に設計して望むアーキテクチャを誘導する
- 品質を下げるのは「賢くないエージェント」より「悪い連携構造」

### Spotify Model
- Squad = 1機能をend-to-endで出す小編成
- Chapter = 全Squad横断の専門レビュア群（security, performance, test）
- Guild = 全体の知識共有規約

### Amazon 2-Pizza Team
- 1タスク3-5エージェントを標準に
- 核心は「小さいこと」より「責任境界が一貫していること」

### Google SRE
- Quality SLO + error budget + toil削減 + blameless postmortem
- 欠陥流出率が予算超過したら新機能生成を制限し、テスト拡充を優先

## 統合: web-globalに適用可能なアイデア

### 即効性が高いもの

| アイデア | 元の分野 | 効果データ | 適用方法 |
|---|---|---|---|
| TDDファースト | SE | リグレッション70%削減 | テスト→実装の順でエージェントに指示 |
| ミューテーションテスト | SE | カバレッジの偽陽性を排除 | CI/スキルに組み込み |
| ブラインドレビュー | 航空CRM/科学 | 確証バイアス排除 | レビューエージェントに実装者情報を隠す |
| 多層防御 | 医療/原子力 | 単一防御は必ず失敗 | lint→型→テスト→レビュー→人間の多段 |
| 即時停止 | トヨタ | 不良品を後工程に流さない | 不確実性検知で生成を中断しユーザーに確認 |
| カスケード | FrugalGPT | コスト最大98%削減 | 簡単な変更はHaiku、複雑な変更はOpus |

### 中期的に検討するもの

| アイデア | 元の分野 | 効果データ | 適用方法 |
|---|---|---|---|
| Best-of-N | AlphaCode | 精度15-40%向上 | 同じタスクを複数エージェントに並列実行（maiで既に実現） |
| セルフプレイ | Sol-Ver | 17-53%改善 | 実装エージェントとテストエージェントの相互改善ループ |
| スキル自動生成 | Voyager/Memento | +13.7pt | 成功パターンをスキルとして自動保存・再利用 |
| SRE/エラーバジェット | Google SRE | 品質予算による自動制御 | 欠陥流出率を計測し閾値超過で新機能停止 |
| 事前宣言 | 科学/DbC | TDDと同根 | コード生成前に期待動作を宣言させる |

### 7原則フレームワーク

1. **即時停止（Jidoka）**: 品質が確認できない状態で先に進まない
2. **多層防御（Swiss Cheese）**: 単一の品質チェックに依存しない
3. **行動多様性（AZdb）**: 異なるアプローチのエージェントを組み合わせる
4. **敵対的検証（Red Team）**: 生成物に対して意図的に問題を探すエージェントを配置
5. **意図明示（Mission Command）**: 「何を達成するか」を明確にし、「どう」は委ねる
6. **構造化チェックリスト（Surgical Checklist）**: 各フェーズで見落としやすい項目を確認
7. **段階的権限（Nuclear Safety）**: リスクに応じて自動実行/確認/人間承認を使い分け

## 9. 敵対的テスト（Adversarial Testing）

### 弁証法的オートコーディング（Block社）
- 2つのAIエージェントが**対立的かつ協調的**にコード生成・検証
- 単純なレビューではなく、設計段階から弁証法的に対話する

### 数値データハイライト
- AI生成コードの脆弱性: 人間コードの**2.74倍**（Veracode 2025）
- 多層防御でプロンプトインジェクション成功率: **73.2% → 8.7%**に削減
- AIコードレビューツール最高検出率: **82%**（Greptile）、誤検出率**3%未満**
- Xbow: **1,400+のゼロデイ**発見、手動の**80倍高速**
- Claudeが2週間でFirefoxの**22件の脆弱性**発見（14件が高深刻度）

### プロパティベーステスト（敵対的テストとしての側面）
- Agentic PBT: 生成バグレポートの**56%が有効なバグ**
- 従来テスト比で**3倍のバグ**を発見

### ファジング
- CKGFuzzer: カバレッジ**8.73%向上**、手動レビュー**84.4%削減**、実バグ**11件**検出
- AutoBug（OOPSLA 2025）: LLMベースシンボリック実行エンジン

### セルフプレイ型敵対テスト
- Sol-Ver: 実装AI × テストAIの対戦で両方が進化
- GASP: 非ガイド型セルフプレイを全基準で上回る

### OWASP Agentic AI Top 10（2026年版）
主要リスク: 目標乗っ取り、ツール悪用、権限不正利用、サプライチェーン脆弱性、予期しないコード実行、メモリ汚染、エージェント間通信の不備、連鎖的障害、信頼の悪用、不正エージェント
