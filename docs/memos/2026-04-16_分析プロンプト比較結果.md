## C が低評価になった原因分析

### Findings で低い理由

1. **Root Cause が FORGOT に偏る（26/70 = 37%）**: 同じ問題を A/B は `READ_BUT_IGNORED` と分類するところを C は `FORGOT` にする。採点者は「FORGOT は対策の方向性が導けない（ルール強化 vs 新設 の判断ができない）」と評価。ただし C の FORGOT は Detail に「なぜ忘れたか」まで書いており、決定木に沿った分類としては論理的に正しい。
2. **証拠の引用が A/B より粗い**: 行番号はあるが、A/B ほど thinking の内容を直接引用していない。
3. **重要度の高い問題を見逃す傾向**: findings 数は 5-6 で A/B と同等だが、テスト未作成・next.config.ts 等のインパクトが大きい問題を落として LOW の問題を拾う傾向。

### Rules で低い理由

1. **数が少ない（39 vs A:47, B:54）**: 1体あたり 3.0 で最少。
2. **Detection が「確認せよ」止まり**: A の `grep -r "/old-path"` や B の `@touchstart があるか` のような具体的なコマンド・パターンが少ない。

### 低評価の根本原因

**決定木の手順を忠実にたどることに集中しすぎて、問題の深掘りが浅くなった。**

- A/B は自由度が高い分、エージェントが「これは重要だ」と判断した問題に時間をかけられる
- C は全問題に対して同じ手順（2-1 → 2-2 → 2-3）を踏むので、重要な問題も些末な問題も同じ深さで処理される
- 結果: A/B は重要問題の Evidence が厚くルールも具体的、C は全問題が均一に浅い

**「構造化しすぎると質が下がる」** という知見。ある程度の自由度を残した方がいい。

### C の良かった点（活かすべき要素）

- 決定木の考え方自体は論理的に正しい。NO_RULE → NOT_READ → MISUNDERSTAND/FORGOT の順で絞り込む設計は対策の方向性を導くのに有用
- 独自の着眼点: 法的文書の原典確認、広告スタブの YAGNI 境界線、DOM二重化のチェックリスト矛盾
- 誤検知が少ない体もある（pr74-1 で正確性5点）

### 改善の方向性

C の決定木を A の自由分析スタイルに **参考情報として組み込む**（強制手順ではなくガイドラインとして渡す）のが最適解の可能性がある。Prompt D として検証する価値あり。

## Prompt D の設計方針

**正確性を重視し、A をベースにする。** 間違ったルールは害になるが、足りないルールは後から足せる。

### A の弱点と改善

**弱点1: 網羅性が低い（3.92 vs B:4.62）**

Step 1 の4基準が「コード品質」に偏っている。エージェントの行動の問題を検出する観点がない。

→ Step 1 に追加:
```
5. エージェントの行動の問題: 不要なファイルのコミット、他ソースからの無検証コピー、
   hook でブロックされた操作の回避、作業ファイル（checklist.md等）の混入
```

**弱点2: INSUFFICIENT_IMPL が曖昧（22/72 = 31%）**

INSUFFICIENT_IMPL は「理解したが実装が不十分」で、対策の方向性が導けない。

→ INSUFFICIENT_IMPL を削除し、FORGOT を追加（C の決定木から取り入れ）。Detail に理由必須。

**弱点3: ルール実用性が B に負ける（4.15 vs 4.62）**

A のルールは Detection が具体的だが、B より数が少なく汎用性でも負ける。

→ Detection の指示を強化: 「Detection には grep パターンやコマンド例を必ず含めること」を追加

### A から保持するもの

- 3段階の分析手順（Step 1→2→3）
- 4基準での問題検出（実PR差異 + ルール違反 + 要件未達 + 設計方針違反）
- session.jsonl の行番号引用を求める指示
- findings テンプレートへの書き込み

### B/C から取り入れるもの

- B: エージェント行動パターンの検出観点
- C: FORGOT の Root Cause 分類（ただしガイドラインとして。強制手順にはしない）
- C: Root Cause 決定木を参考情報として添付

## 最終結論

**Prompt A を採用。**

理由:
- 正確性と証拠の具体性で一貫して最高
- 余計な指示を足しても改善しないことが D で検証済み
- ルール品質（横断評価）も最高（4.5/5）

### ESLint カスタムルール化の効果分析

**セッションログを分析した結果、ESLint カスタムルール化の効果はなかった。むしろ逆効果。**

| 事実 | 詳細 |
|---|---|

| lint 実行回数 | 設定2: 7回 → 設定3: 23回（増分は既存ファイルの誤検知対応） |

### +1.0 の真の原因

ESLint ではなく、**ルール数を 112→104 に絞ったことで基本ルール（i18n 等）が埋もれなくなった効果**。

| 観点 | 全部入り | lint+絞り | 差 |
|---|---|---|---|
| 多言語 | 3.0 | **3.7** | +0.7（i18n ルールが目立つようになった） |
| テスト | 1.0 | **2.0** | +1.0（AI-TEST-001 を読んで1体が自発的にテスト作成） |
| SEO/YAGNI | **3.7** | 2.7 | -1.0（YAGNI ルールが埋もれた？） |

### 教訓

1. **ESLint ルール追加前に既存コードとの整合性チェックが必要**: dangerouslySetInnerHTML が必須な既存コード（JSON-LD 出力等）に例外設定なしで追加した結果、無駄な修正ループが発生
2. **タスク固有でないルールは該当タスクで実験すべき**: lib/api/ 関連ルールは about ページ実装では発火しない。API 実装タスク（PR59 等）で実験しないと効果が測れない
3. **ルール数を減らすこと自体に効果がある**: 112→104 の削減で i18n 基本ルールの遵守が改善。情報量の最適化が重要
