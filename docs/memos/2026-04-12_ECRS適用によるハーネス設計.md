## E（Eliminate / 排除）

### 判断基準
「これをやめたら何が起きるか？」何も起きないなら排除。
**注意:** 将来使うものは排除しない。「今不要」ではなく「永続的に不要か」で判断。

### 排除対象

| 対象 | 理由 |
|---|---|
| CLAUDE.md内のlintツール紹介（「prettier導入済み」等） | pre-commitで自動実行される。書かなくてもlintが走る |

### 排除しない（検討して残したもの）

| 対象 | 残す理由 |
|---|---|
| adoption-policy.md | Phase B/Cで将来使う |
| frontend-rules/examples/ | ルール違反時の参考になる |
| frontend-rules/references.md | ルール変更・議論時に必要 |
| frontend-rules/web2-migration-component-mapping.md | 移植時に使う |
| CLAUDE.md「CI/CD構築予定」 | 構築時に参照される |
| skills/readme-updater, skills-readme-generator | 管理用途で使う |

### 排除のインパクト
極めて小さい。2行程度の削減。

### 継続的な排除の仕組み: 計測hookによる未使用ファイル特定

PostToolUseのRead matcherでファイルアクセスをログに記録し、実際にどのdocsが使われているかを客観的に計測する。

```json
{
  "PostToolUse": [
    {
      "matcher": "Read",
      "hooks": [
        {
          "type": "command",
          "command": "cat | jq -r '.tool_input.file_path // empty' >> /tmp/agent-file-access.log",
          "async": true
        }
      ]
    }
  ]
}
```

計測結果から分かること:
- 一度も読まれないファイル → 本当の排除候補
- 読んだのにバグが出た → ルールの書き方が悪い or コンテキストに埋もれた
- 読まなかったからバグが出た → rulesのpathsで自動注入すべき

## R（Rearrange / 再配置）

### 判断基準
「順番や場所を変えるだけで効率が上がらないか？」

### 方針: 全部最初に読む → 必要な時に必要なものだけ

| # | 現状 | 再配置後 |
|---|---|---|
| 1 | frontend-rules 2,550行がCLAUDE.md参照で読まれる可能性 | `.claude/rules/` にpaths付きで配置。CSSファイル触った時だけCSS責務ルールが注入 |
| 2 | CLAUDE.mdにアーキテクチャ詳細が常時ロード | `app/` 配下を触る時だけ注入されるruleに |
| 3 | 20スキルが常にスキル一覧に表示 | フェーズで分離（実装用 / レビュー用 / 管理用） |
| 4 | 移植ルールが全ファイル共通で読み込まれる | 移植作業時のみ注入 |

### rules配置案

```
.claude/rules/
  css.md          → paths: ["**/*.css", "**/*.module.css", "app/**/*.tsx"]
  component.md    → paths: ["components/**/*.tsx"]
  page.md         → paths: ["app/**/page.tsx", "app/**/layout.tsx"]
  api.md          → paths: ["lib/api/**/*.ts"]
  i18n.md         → paths: ["messages/**/*.json"]
  seo.md          → paths: ["**/*metadata*", "**/*json-ld*", "**/*sitemap*"]
```

### 結果
CLAUDE.mdは50行以下のシンプルな案内に。具体ルールはファイルを触ったタイミングで自動注入。コンテキスト効率が大幅改善。

## 設定C（トヨタ式）の全体像

ECRSに加えて、以下のトヨタ概念も組み込む:

| 概念 | 実装 |
|---|---|
| ECRS | 上記の排除・結合・再配置・簡素化 |
| 自働化（Jidoka） | PostToolUse hooksで品質チェック。問題検知で停止 |
| ポカヨケ | rulesのpathsで、ファイル種別ごとにルール自動注入。読み忘れが構造的に不可能 |
| アンドン | 計測hookでファイルアクセスをログ。何が読まれているか可視化 |
| 自工程完結 | 各rules/が自分の担当範囲の品質を保証。後工程のレビューに頼らない |
| 標準作業 | CLAUDE.md 30行以下 = 最善の手順を1つに絞る |
| なぜなぜ分析 | バグ発生時に計測ログを遡り「読まなかったから」「読んだが埋もれた」を特定 |
