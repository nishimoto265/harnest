# トヨタ式カイゼンのハーネスエンジニアリング適用

## 調査日
2026-04-11

## カイゼンの本質: 足すのではなく引く

### ECRS（改善の4原則）— 優先順位順

1. **Eliminate（排除）** — まず「やめられないか」
2. **Combine（結合）** — まとめられないか
3. **Rearrange（再配置）** — 順序を変えられないか
4. **Simplify（簡素化）** — もっと単純にできないか

最も効果が高いのは「やめること」。足すのは最後の手段。

### 標準作業の考え方
- ルールを増やすのではなく、**最善の手順を1つに磨き上げる**
- 標準は「守るべき不変のルール」ではなく「改善の基準であり、進化させるもの」
- 大野耐一: 「標準のないところに改善はない」= ルールをたくさん作れではなく、基準を明確にせよ

### 7つのムダ — 特に「加工のムダ」に注意
- **加工のムダ**: 要求以上の品質・機能のために過剰な手間をかけること
- 不要なルールが1つ増えると、確認作業・報告作業・管理作業が連鎖的に発生
- **つくりすぎのムダが最悪** — 他の全てのムダを誘発する

### 自工程完結
- 後工程に不良を流さない
- **ルールを足して後から検査するのではなく、工程自体の質を上げてルールを不要にする**
- 起源: 豊田佐吉の「糸が切れたら自動で止まる織機」— 検査が不要になる仕組み

### カイゼンの「やりすぎ」問題
- 完璧を求めすぎると不要な変更が増える
- 手順書が多すぎると形骸化して使われない
- 改善提案のノルマ化 → 本質的でない「改善のための改善」が発生
- 大野耐一の鬼十訓第1条: 「君はコストだ。まずムダを削れ」

### トヨタが実際にルールを減らした事例
- 誰も読まない報告書の廃止
- 不要な検査項目の削除
- 堤工場: 完成車の雨漏れ検査を廃止 → 製造工程自体で品質を作り込み検査を不要にした

## ハーネスエンジニアリングへの適用

### ❌ アンチパターン: ルール追加の連鎖
```
ミス発生 → ルール追加 → ミス発生 → ルール追加 → 肥大化 → 形骸化
```

### ✅ ECRS適用パターン
```
ミス発生 →
  1. Eliminate: そのチェック自体をなくせないか（テスト・型で自動化して人/AIの判断を不要に）
  2. Combine: 既存のルールと統合できないか
  3. Rearrange: rulesのpathsで適切なタイミングに変えれば防げないか
  4. Simplify: どうしても必要なら最も単純な形で追加
```

### 自工程完結の適用
- 「余白の責務を間違える」→ ルールを追加するのではなく、コンポーネントの設計パターン自体を改善して間違えようがない構造にする
- 「PRテンプレートに従わない」→ hookでブロックするのではなく、PRスキル自体がテンプレートを読む設計にする

### PDCAで回す（足すだけでなく削る）
```
Plan: ミスの原因を分析
Do:   対策を1つ試す（できればコードで制御）
Check: 効果があったか確認
Act:  効果あり → 標準化
      効果なし → やめる
      既存ルールが不要になった → 削除
```

### 定期棚卸し
- CLAUDE.md / rules / hooks / skills を定期的にレビュー
- 効いていないルールは削除
- 重複しているルールは統合
- コードで制御可能になったルールはlint/テストに移行してドキュメントから削除

## 判断基準: ルールを追加すべきか

| 質問 | Yes → | No → |
|---|---|---|
| テスト・型・lintで防げる？ | コードで制御。ルール不要 | 次へ |
| 既存ルールと統合できる？ | 統合。新規追加しない | 次へ |
| pathsで必要な時だけ読ませれば十分？ | rulesに配置 | 次へ |
| 全タスクで毎回必要？ | CLAUDE.mdに1行追加 | 追加しない |
