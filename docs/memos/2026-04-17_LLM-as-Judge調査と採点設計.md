# LLM-as-Judge 調査と採点設計

## 背景

ハーネスのルールライフサイクル（お試し→本採用→削除）を全自動で回すために、AI による品質評価（compliance-check）の信頼性を確認する必要がある。

## LLM-as-Judge の信頼性データ

### 人間との一致率

| Judge モデル | 人間との一致率 | 出典 |
|---|---|---|
| GPT-4 | ~80% | Zheng et al. (2023) MT-Bench |
| GPT-3.5 | ~65% | 同上 |
| Claude系 | 自己バイアスが比較的低い | Panickssery et al. (2024) |
| 人間同士 | ~81% | ベースライン |
| Prometheus (ルーブリック付き) | Pearson相関 0.89 | Kim et al. (2024) |

**結論**: 強いモデル + ルーブリックで人間同士の一致率に近い精度が出る。

### 既知のバイアス

| バイアス | 内容 | 影響 | 対策 |
|---|---|---|---|
| Position bias | 最初の回答を好む(~60%) | pairwise比較時 | 順序入れ替え2回評価 |
| Self-enhancement | 自分のモデルを~10pt高く評価 | 自己評価時 | 異なるモデルファミリーで評価 |
| Verbosity bias | 長い回答を~70%で好む | direct scoring時 | ルーブリックで明示抑制 |
| Authority bias | 自信に満ちた文体を好む | 全般 | ルーブリックで内容重視を明示 |

### 信頼性を上げる方法

| 方法 | 効果 | コスト |
|---|---|---|
| ルーブリック明確化 | +10pt一致率向上 | 低（1回作れば再利用） |
| CoT理由付き | 一貫性+15%向上 | 低（プロンプト変更のみ） |
| 複数judge合議 | +5-8pt | 中（コスト3倍） |
| Reference-based | +5-10pt | 低（参照回答があれば） |
| Pairwise比較 | +5-8pt | 高（O(n²)比較） |

### コードレビューでのLLM評価

| 領域 | LLM | 人間 |
|---|---|---|
| 命名・スタイル違反 | ◎ 網羅的で一貫 | ○ |
| 既知バグパターン | ○ 同等 | ○ |
| 可読性・コメント品質 | ○ lintでは不可能 | ○ |
| ビジネスロジックの誤り | △ | ◎ |
| アーキテクチャ判断 | △ 局所的 | ◎ |

出典: CodeRabbit の指摘の約60-70%が有用(actionable)、false positive率は約20-30%。

## 確率論的改善の妥当性

### Condorcetの陪審定理

個々の判定の正答率が50%超なら、独立した判定の多数決で正しい判定に収束する。

| judge精度 | 5回多数決 | 10回多数決 |
|---|---|---|
| 60% | 68% | 75% |
| 70% | 84% | 91% |
| 80% | 94% | 98% |

### 重要な条件

- バイアスが系統的（同じ方向に偏る）場合は収束しない
- **「評価が微妙なら変更しない」デフォルト**が必要
- 改善に確信がある時だけ変更、迷ったら現状維持

### 実用的な精度の閾値

| 精度 | 用途 |
|---|---|
| 60-65% | 傾向分析、A/Bテスト。個別判定は不安定 |
| 70-75% | 自動パイプラインのフィルタリング |
| 80%以上 | 人間レビューの大部分を代替可能 |

### Noisy Labels 理論

- ノイズ率50%未満なら十分なデータ量で真の分布を復元可能 (Natarajan et al. 2013)
- 精度60%のjudgeでも10回の多数決で誤採用確率は2%未満（二項分布）

## 我々の採点への適用

### 現状の課題

今回の実験では:
- 異なるサブエージェントが異なる条件を採点 → 採点者間のキャリブレーションなし
- 5観点×5点のdirect scoring → ばらつきが大きい
- ベースラインpr74-2(23点)のような外れ値が発生

### 信頼性向上のために適用すべき手法

| 手法 | 現状 | 改善案 |
|---|---|---|
| ルーブリック | 「1-5で採点」のみ | 各観点・各スコアの具体的定義を明文化 |
| CoT理由付き | あり（理由付き採点） | 維持 |
| Reference-based | 実PR patchを提供 | 維持（これが強み） |
| 複数judge | 1体で採点 | 3体で採点→多数決 or 平均 |
| 採点者統一 | 条件ごとに別エージェント | 全条件を1体が採点 or 同一プロンプトを徹底 |
| Pairwise | 未使用 | 「A条件 vs B条件、どちらが良いか」で比較 |

### 改善された採点プロンプト設計

```
## ルーブリック（各観点の定義）

### コンポーネント分割 (1-5)
1: page.tsxに全て集約。components/に分離なし
2: components/に分離あるが、1-2ファイルのみ
3: セクション単位で分離あるが、共通化/階層化なし
4: organisms/molecules相当の分離。実PRと同等の構成
5: 実PRと同等 + テストファイル付き

### CSS実装 (1-5)
1: box-shadowをTailwind arbitrary valueでインライン展開
2: 一部CSS Modulesだがインラインも混在
3: CSS Modulesに集約。CSS変数なし
4: CSS Modules + 適切なbreakpoint。実PRに近い
5: CSS Modules + CSS変数 + 全breakpoint正確。実PRと同等

[...各観点を同様に定義]

## 採点手順
1. 実PRのdiffを読む
2. エージェントのdiffを読む
3. 各観点についてルーブリックに照らして理由を先に書く
4. 理由に基づいてスコアを付ける
```

### 採点の二層構造

```
Layer 1: 自動検出（grep/lint）
  - 確実に判定できる項目を先に機械的に採点
  - 例: twitter追加あり → SEO -1点 (確定)
  - 例: CSS Modules不使用 → CSS 1-2点 (確定)

Layer 2: LLM評価（ルーブリック付き）
  - Layer 1で判定できない項目をLLMが採点
  - ルーブリック + CoT + reference-based
  - 信頼性の高い条件が揃っている
```

## テスト/lint vs LLM評価の使い分け

| 品質の側面 | テスト/lint | LLM評価 | 採用 |
|---|---|---|---|
| 構文エラー | ◎ | 不要 | テスト/lint |
| 型安全性 | ◎ | 不要 | テスト/lint |
| スタイル一貫性 | ◎ | 不要 | テスト/lint |
| 可読性・保守性 | × | ○ | LLM |
| 設計の妥当性 | × | △ | LLM（局所的） |
| ルール遵守判定 | △（一部grep可） | ○ | 併用 |
| アーキテクチャ | × | △ | 人間（高ステークス時） |

**原則: テスト/lintで検出可能な問題はLLMに評価させない。LLMは「テストでは検出不能だが品質に影響する」領域に集中させる。**

## 実運用での採点方式: 絶対評価 + Pairwise並行

ハーネス改善時（ルール追加/変更/削除）は、**絶対評価とPairwise比較を並行して実施**する。

### 実施フロー

```
現在のbest設定でPR実装（3体）
新しい設定でPR実装（3体）
  ↓ 並行で2種類の採点を実施
  ├─ 絶対評価: 各体を scoring-rubric.md で独立採点（5観点×5段階）
  └─ Pairwise: best体 vs new体 を5観点で比較判定
  ↓
両方の結果を突合して判断
```

### 絶対評価（継続的なスコア蓄積用）

- 全PRの全体に対して実施し、スコアを蓄積
- 経時変化の追跡が可能（スコアの推移グラフ等）
- ルーブリック: `docs/harness-eval/scoring-rubric.md` (Layer1 + Layer2)
- 1体1エージェントで採点

### Pairwise比較（改善判定用）

- best vs new の2条件比較に特化
- 絶対スコアより安定（+5-8pt一致率向上）
- 判定が単純（A勝ち / 引き分け / B勝ち）

```
プロンプト:
  以下の2つのエージェント出力を比較してください。
  参照実装と5観点のルーブリックに基づき、各観点で
  A勝ち / 引き分け / B勝ち を判定してください。

  参照実装: @actual-pr.patch
  ルーブリック: @scoring-rubric.md
  A (best): @best-diff.patch
  B (new): @new-diff.patch

  各観点: 理由(2-3文) → A / Draw / B
  最後に勝敗サマリ: A:N Draw:N B:N
```

### 判定基準

3体分のPairwise結果(5観点×3体=15判定)で:
- **Bの勝ち越しが8以上** → 新設定を採用（bestを更新）
- **差が2以内** → 有意差なし。コスト/複雑さが低い方を採用
- **Aの勝ち越しが8以上** → 新設定を棄却

絶対評価のスコアも確認し、Pairwiseと矛盾する場合は:
- 絶対スコアの平均差が1.0以上あればそちらを優先
- 両方一致していれば確信度が高い

### 使い分け

| 場面 | 方式 | 理由 |
|---|---|---|
| ハーネス改善時(A/B比較) | Pairwise + 絶対評価 | 相対比較の安定性 + 絶対値の蓄積 |
| 定期モニタリング | 絶対評価のみ | 経時変化の追跡 |
| ルールのお試し→本採用判定 | compliance-check | 遵守/違反の二値判定で十分 |

## 複数judge合議: パネルレビュー方式

### アーキテクチャ

単純な多数決/平均ではなく、メインClaudeが裁定するパネルレビュー方式を採用する。

```
採点対象 diff + ルーブリック + 参照PR
  ├─ Claude (sonnet) → スコア + 理由JSON
  └─ Codex (OpenAI)  → スコア + 理由JSON
  ↓
メインClaude（オーケストレーター）
  - 両方のスコアと理由を読む
  - 理由の根拠の強さを比較
  - 矛盾がある場合は根拠が強い方を採用、または独自に最終判断
  ↓
最終スコア + 判断理由
```

### 単純合議との違い

| 方式 | メリット | デメリット |
|---|---|---|
| 平均/多数決 | 実装が簡単 | 理由の質を無視。弱い根拠でも同じ重み |
| パネルレビュー | 理由の質で判断。矛盾自体が情報になる | メインClaudeのコスト追加 |

### メインClaudeの裁定基準

1. **2体が一致（差1以内）** → そのまま採用。裁定不要
2. **2体が矛盾（差2以上）** → 両方の理由を読み:
   - 具体的な証拠（diff行番号、grepコマンド結果等）がある方を優先
   - プロジェクト文脈（ルーブリックのLayer2）に照らして妥当な方を採用
   - 判断理由を記録（「Codexはbreakpointのズレを見逃していたため、Claudeの2点を採用」等）
3. **どちらの理由も不十分** → メインClaude自身がdiffを確認して独自採点

### 異なるモデルファミリーを使う理由

- **self-enhancement bias排除**: 同一モデルは自身の出力を高く評価する傾向（~10pt）がある
- Claude + Codex(OpenAI) = 異なるファミリーでバイアスが打ち消し合う
- 2体で十分（3体合議のコスト増なしに信頼性向上）

### 実装

```bash
# 並列で2体に採点させる
claude -p "$SCORING_PROMPT" --model sonnet > score_claude.json &
codex -p "$SCORING_PROMPT" > score_codex.json &
wait

# メインClaudeが裁定
claude -p "以下の2つの採点結果を読み、最終スコアを決定してください。
Claude判定: @score_claude.json
Codex判定: @score_codex.json
ルーブリック: @scoring-rubric.md
矛盾がある場合は理由の根拠の強さで判断。" > final_score.json
```
