# 発表資料 構成案: Harness 自動生成システム

- 発表時間: 7 分（2 人で分担、1 人あたり 3 分強）
- 聞き手: 社内エンジニア
- 目的: システムの紹介 + 知見の共有
- 時間配分は目安。盛りすぎず、各セクションの「重要な点」を大まかに伝える方針

---

## 1. 背景（〜1 分）

- **harness を人手で作るのは煩雑**: 抜け漏れ / 属人化（書いた人しか分からない）/ 陳腐化（古い rule が残り続ける）
- **知見の共有がしたい**:
  - 各自でスキルを習得することは可能だが、それをチームで共有したい
  - 会話ログや PR 内容から自動でスキルを抽出し、harness を作る仕組みを作りたい
  - 同じ課題で何度も躓かないようナレッジ化したい
- （任意・1 言）harness engineering（Martin Fowler）の考え方に基づく: **Agent = Model + Harness**。流行りでなく体系に沿った取り組み

---

## 2. 実施内容（システム構成とワークフロー）（〜2.5 分）

### (a) エージェント構成
- Databricks / GitHub / MCP（Model Context Protocol）を活用
- 3〜4 体のエージェント（main / classifier / review / self-check）を、役割だけ図で配置して紹介（中身の詳細は省く）

### (b) データ取得 + スキル作成 ← ここは厚めに
- main agent が GitHub（PR 本文・diff・レビューコメント）と Databricks（既存 rule）に接続してデータ取得（図 + 矢印で）
- そこから「繰り返し指摘されそうなパターン」を抽出 → classifier で分類（新規/更新/重複/却下済み/統合）→ review で審査 → rule（スキル）化
- 「どういう流れでスキルが作られるか」をフローで見せる

### (c) PR と承認フロー
- スキルができたら **1 rule = 1 PR** で提出 → 人がレビュー
- 内容が良ければ approve してマージ、ダメなら close（理由コメント付き）

### (d) tier システム
- 観測されるたびに tier が上がり、**tier=3 になったものだけ配布**
- 「何度も繰り返し出た重要なものだけが配られる」= ノイズを除外する仕組み

### (e) self-check（効果測定）← 差別化ポイント
- 配った rule が**実際に守られたか**を、会話ログ + PR から評価（compliant / violation / n_a / exception）
- 守られていない rule は見直し → **配って終わりにせず、改善し続けるループ**
- 例: レシピ画像配列 Skill を実装セッションに適用し、動作を壊す致命的問題が見つかったら削除 PR を改善材料として上げる
- 「同じ轍を踏まない」を実現する核

## 3. 実施結果（〜1.5 分）

- 実際に作成された **PR のスクリーンショット**
- **削除（archive）の仕組み**の紹介
- **GitHub Actions で自動 trigger を実証**（PR をトリガーに agent が起動 → スキル作成まで）

---

## 4. 知見と課題 / 今後（〜1.5 分）

### (a) 課題: ワークフロー形式には向きづらい
- Managed Agents 内部にスクリプト実行環境がないため、ワークフロー形式での運用にはあまり向いていない

### (b) 良い点: 実行環境を任せられる
- 外部から起動することができれば、クラウド上でエージェントを動かすことが可能

### (c) 今後の展望
- **対象を広げる**: CLAUDE.md などの対象を広げ、ドキュメントとの内容のズレを検知できるようにする
- **自動運用**: PR のマージを起点とした自動運用を導入する
- **Outcomes の活用**: Managed Agents の実行結果を測定する仕組みである Outcomes を活用し、生成システム自体を改善していく

---

## 6. プラグイン（導入のお願い）（〜1 分）

### (a) 課題と経緯
- データ収集の際に「**この PR を書いた Claude Code セッションの会話**」を結び付ける必要がある。だが PR ↔ session_id の紐付けは Claude Code 標準では取れない
- そこで紐付け情報を集める Claude Code 用の **harness-plugin** を作った（hook 形式で動く軽量プラグイン）
- これがあると「どのセッションがどの PR の作業か」が DB で引けるようになり、Harness Agent に渡せる情報量が増え、より良いスキル作成につながる

### (b) 将来配布・導入のお願い
- 近い将来、Claude Code 用 plugin として公開予定
- 公開した際は、**Claude Code を使う社内開発者の手元への導入に協力してほしい**（人数が増えるほど self-check のデータが充実し、システム全体の精度が上がる）
- 導入は軽量（Claude Code の plugin として hook を仕込むだけ）

### (c) 取得する情報（ここをしっかり説明）
plugin が Databricks に送るのは **紐付け情報だけ** です：

| 項目 | 例 |
|---|---|
| `session_id` | Claude Code のセッション ID |
| `git_remote_origin` | repo の URL |
| `git_branch` | branch 名 |
| `pr_number` | PR 番号 |
| `event_timestamp` | 時刻 |

**会話本文・コード内容・個人情報は plugin では一切取りません**。会話ログ自体は Claude Code 標準のテレメトリで（plugin とは別経路で）既に組織が収集しており、plugin はその「**どのセッションが、どの PR の作業だったか**」のメタ情報だけを足します。
→ プライバシーに踏み込む新規の情報収集ではないので、安心して入れてほしいです。
