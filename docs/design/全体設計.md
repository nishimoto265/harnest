# 自動改善パイプライン全体設計

## 目的

ハーネス（ルール群・採点基準・実行環境）を、人間の判断を極力挟まずに継続改善する自動フローを定める。
個別実装（スクリプト・プロンプト）は後で差し替えられる構造にし、まず全体の骨格を固める。

## 設計原則

1. **アーティファクト駆動** — ステップ間は必ずファイル(jsonl/md)で繋ぐ。メモリ内状態を持たない
2. **番号付きステップ** — UNIX的に `10- / 20- / ...` で順序とフェーズ境界を可読化
3. **実装と契約の分離** — ステップは I/O スキーマを固定し、中身(プロンプト/モデル/ロジック)は config で差し替え
4. **1PR完結** — バッチ学習ではなく、PRごとに完結する前後比較サイクル

---

## 全体構造

```
サイクル② (1PRごと・主サイクル)
  パス1実装 → 採点 → ルール候補 → パス2実装 → 前後比較 → 採用/棄却
  
サイクル③ (累積判定・月次 or N件トリガー)
  既存ルールの status 遷移(サンセット)
  重複ルールの統合
```

compliance監査はサイクル②の採点に内包されるため、別サイクル(旧①)は不要。

---

## サイクル② : 1PR完結の前後比較サイクル

### 前提

- 1PR = 1サイクル
- 学習源と検証対象が**同じPR**
- 比較軸は「現行best設定での実装(パス1)」

### フロー

```
[検知] マージPRの差分検出 (state.json で冪等管理)
   │
   ▼
[step 10] ベース状態復元
   そのPRの実装開始時点のブランチ状態を再現
   → worktree 3個分の作業ディレクトリ準備
   │
   ▼
[step 20] パス1: best設定で実装
   - 現行ルール全適用 + 同タスク指示
   - worktree × 3体 並列実装
   - 成果物 A1, A2, A3
   │
   ▼
[step 30] パス1 採点
   評価軸:
     1. ルーブリック5次元採点 (Correctness/Design/Idiomatic/Fidelity/Discipline)
     2. 実PR実装との差分 (Fidelity の根拠)
     3. Discipline内に compliance監査を内包
        (checklist-result.md を読み、[x]/[!]/[?] の妥当性を判定)
   出力: scores.jsonl, compliance.jsonl(パス1分)
   │
   ▼
[step 40] ルール候補抽出
   パス1の問題点 (採点理由・実PR差分・未準拠項目)
     ↓ AI抽出(Claude)
   候補ルール群 (new / update / duplicate に分類)
     - new:       既存にない → そのまま候補
     - update:    既存ルールの具体化 → 差し替え案扱い
     - duplicate: 既存と同義 → 破棄
   │
   ▼
[step 50] パス2: 候補ルール追加版で実装
   - 同じベース状態を worktree × 3体 で再現
   - best設定 + 候補ルール(全部入れ)
   - 同タスクを実装
   - 成果物 B1, B2, B3
   │
   ▼
[step 60] パス2採点 + 前後比較
   1. 5次元スコア: パス1 vs パス2 の差分
   2. Pairwise: A_i vs B_i (同エージェント・同PR・設定違い)
   3. Fidelity改善度: 実PRとの距離の減少幅
   │
   ▼
[step 70] 採用判定
   改善 → 採用: rules.md に追加 / best設定を更新
   横ばい → 保留(保留キューへ。次PRで再評価機会)
   悪化 → 棄却: archived/rejected/ に保存(消さない)
   │
   ▼
state.json の last_processed_pr を更新 → 次PRへ
```

### ステップ I/O 契約

| step | 入力 | 出力 |
|---|---|---|
| 10 | PR番号 | worktree 3個 + base_sha 記録 |
| 20 | worktree + best-rules + タスク指示 | 成果物A×3 + checklist-result×3 |
| 30 | 成果物A + 実PR diff + rubric | scores.jsonl(A), compliance.jsonl(A) |
| 40 | 採点結果 + 既存ルール群 | candidate-rules.md (new/update) |
| 50 | worktree(再作成) + best+候補 | 成果物B×3 + checklist-result×3 |
| 60 | 成果物A, B + 実PR | scores.jsonl(B), pairwise.jsonl |
| 70 | 全評価結果 | decision.json + rules-next.md (採用時のみ) |

## 採点の役割 (3つ)

採点は単なる評価ではなく、下流の意思決定データ源。

| 採点タイプ | 使い道 |
|---|---|
| 絶対評価(5次元) | 改善幅測定(パス1 vs パス2) |
| Pairwise | 採用判定の主シグナル(同エージェント間で勝敗) |
| compliance監査(Discipline内包) | サンセット判定のデータ蓄積 |

### ジャッジ構成 (Panel Review)

```
Claude判定   ┐
Codex判定    ├─► メインClaudeが仲裁 → 最終スコア
             ┘                         + 理由欄
```

メインClaudeは理由の質で判断(片方が根拠希薄なら採用しない)。

### checklist 3記号

エージェントが自己申告で埋める。「違反した」は書かせない(観測と評価を分離):

| 記号 | 意味 |
|---|---|
| `[x]` | 準拠(該当し、守った) |
| `[-]` | 対象外(該当する場面がなかった) |
| `[?]` | 例外判断(該当領域だが、このケースには当てはまらないと判断。理由必須) |
| `[ ]` | 未記入(抜け) — 別トラックで品質指標化 |

違反検出は**採点ジャッジ(step 30)が統合して実施**。別ジャッジを呼ばない。

## サイクル③ : 累積サンセット判定

### 起動契機

```
(新規処理PR ≥ 10件) AND (前回から ≥ 2週間)
  OR
(前回から ≥ 3ヶ月)  ← 時間セーフティ
```

1PR単発では統計的に判定不可能な「長期の形骸化」を検出する。

### 判定ロジック (フェーズ依存)

| フェーズ | 主シグナル | at-risk 最低発火数 | 段階数 |
|---|---|---|---|
| **ブートストラップ期(今)** | 寄与度(パス1 vs パス2差) | 1-2 | 2遷移で削除 |
| 安定運用期 | compliance率の累積 | 5 | 2-3遷移で削除 |

### status 3段階

```
active ──(悪い兆候)──► at-risk ──(2回目の悪化)──► removed
                          │
                          └──(回復)──► active
```

削除ルートを3種:

| 経路 | 条件 | 認定回数 |
|---|---|---|
| 即削除 | Δ < -0.3 (ルール入れたら明確悪化) | 1回 |
| 2回確認削除 | compliance低 or 寄与度ゼロ付近 | 2回 |
| 不発削除 | 発火0が連続3ゲート | 別ルート |

### removed の扱い

ファイル削除ではなく `archived/removed/` へ移動。フロントマターに履歴を残す:
- 削除理由(compliance率/寄与度/未発火)
- 削除時の採点データ
- 復活判断の根拠になる情報

## ルール棄却時の保存 (archived/)

サイクル②で棄却されたものも消さない。理由:
- 次のPRで同じ提案が出たら再評価材料
- 「なぜこの方向を取らないか」の歴史記録
- 将来のより賢いモデルで再評価可能

```
docs/ai-rules/
  generated-rules.md          # 現行採用ルール(status: active/at-risk)
  archived/
    rejected/                 # サイクル②で採用されなかった候補
      2026-04-17_PR74_*.md
    removed/                  # 一度採用後、サンセットされたもの
      2026-05-01_*.md
```

## フェーズ 2段階

PR累積数に応じて判定基準を切り替える:

| フェーズ | 条件 | 判定の主軸 |
|---|---|---|
| **ブートストラップ期** | 累計 1〜20PR | 寄与度(1PR内の前後比較) |
| 安定運用期 | 30PR〜 | compliance率/usage率の統計判定 |

今は完全にブートストラップ期。統計的判定は期待しない。

### 保護ガード

- 採用後の最初1サイクルは `removed` 禁止
- ルール発火0は判定対象外
- 連続起動禁止: 前回サイクル③から最低2週間

```
生成された候補ルール群
         │
         ▼ AI分類(既存ルール群と照合)
  ┌──────┼──────┐
  ▼      ▼      ▼
 new    update  duplicate
  │      │       │
  │      │       └─► 破棄
  │      │
  │      └─► 既存を差し替え対象として検証
  │           (baseline=既存表記, 候補=更新表記)
  │
  └─► 純粋な新規追加として検証
```

## 検知とトリガー

### サイクル② 起動

PRマージの検知は**差分検知ポーリング**方式:

```bash
state_file=pipeline/state.json
last=$(jq -r .last_processed_pr $state_file)

new_prs=$(gh pr list --state merged --limit 20 \
          --json number,mergedAt,title \
          --jq "[.[] | select(.number > $last)]")
```

- 手動起動でもcron化でもOK(初期は手動)
- state.json で冪等性担保
- 再実行可能(stateを巻き戻せば任意地点から)

GitHub Actions / Webhook は当面不要(一人運用なら過剰)。

## 3体の差別化方針

Claude Code では乱数シードを明示設定できない(Temperature等を外から固定する手段が薄い)。
→ **同一 CLAUDE.md / 同一プロンプトで素直に3回走らせる**。LLMの非決定性で自然に分散する。

補足:
- 3体の結果がほぼ同じ = その設定下でのルール効果が安定している証拠(むしろ好ましい)
- 分散が過大なら、ルール記述の曖昧性として扱う

## パス2のルール適用方針

候補ルール全部入れで一括再実装する。**1本ずつの個別検証はしない**。

| 理由 | 内容 |
|---|---|
| コスト | 個別検証はルール数倍の実験回数 |
| 目的 | 1PR完結サイクルは「このPRで問題になった点」のセットとして扱う |
| 悪化時の対応 | 全部棄却(archived/rejected/) で十分 |

将来、全部入れで悪化したときの**切り分けロジック**が必要になった時点で追加検討する
(現状は発生件数が見えないので設計しない)。

## Fidelity 採点の方針

「実PR との差(diff)を埋める」ではなく、「**実PRを参考資料として採点時に提示する**」。

理由:
- 同じ要件に対して正しい実装パターンは複数ある(書き方は自由)
- diff の一致を強制すると、型が違うだけで減点される
- 採点者(ジャッジLLM)が「趣旨として合っているか」を判断する方が妥当

実装:
- 採点プロンプトに「参考: このPRで実際にマージされた実装は以下」セクションを追加
- diff は貼るが、「一致を求めるものではない。意図・構造の参考として使え」と明示
- Fidelity次元の採点基準: 「実PRと趣旨が一致しているか」で測る

## PR単位の並列処理方針

**PR間は完全直列(queue処理)、1PR内のagent 3体は並列** という構成。

### 構造

```
PR Queue:
  PR#74 → [pass1(3体並列)→採点→抽出→pass2(3体並列)→比較→採用] → 完了
    ↓ (完了してから次を始める)
  PR#75 → [同上]
    ↓
  PR#76 → [同上]
```

### 理由

Codexレビュー(3体並列)でFIFO stale問題を3体全員が指摘。PR間並列化は:
- 古いbestでの検証になる(先行採用を追従できない)
- 重複候補をskipする無駄
- スナップショット管理/lock/compare-and-swapの実装複雑性

を抱えるため、PR間直列化ですべて解消する。

### 時間見積もり

- 1PRあたり ≈ 1時間(agent 3体並列込み)
- 週5PR想定なら累計5時間 → 夜間/週末バッチで十分
- 並列のメリット(速度)は規模次第で後日検討

### worktree構造

```
runs/<date>-PR<number>/
  20-pass1/a1/  a2/  a3/   (agent 3体並列、互いに独立)
  50-pass2/a1/  a2/  a3/
```

agent 3体は独立worktreeで並列実行、PR間は直列。

## このフローが解決する問題

| 課題 | 解決方式 |
|---|---|
| ルール生成の偏り | 1PRごとに実PR実装と比較して汎化 |
| ルール過多による認知負荷 | サンセット+保護ガードで自然淘汰 |
| 採点の変動 | 同ベース・同タスク・設定違いで変数統制 |
| 「違反」自己申告のバイアス | 記号を3択に制限し、違反検出はジャッジ側に分離 |
| 削除判断の難しさ | status 3段階+段階遷移+archived退避で取り消し可 |
| コスト爆発 | checklist自己申告+発火したルールのみ監査 |
| 学習源PR収集の遅さ | 1PR完結にしたので累積不要 |
