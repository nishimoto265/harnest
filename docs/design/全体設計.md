# 自動改善パイプライン全体設計

この文書は、現在の Go 実装と `internal/contracts/**` に同期した全体像である。
field 単位の schema、step70 staged transaction、recover / resume の厳密な state machine は
以下を正本とする。

- `docs/design/io-contracts.md`
- `cmd/auto-improve/**`, `internal/**`, `scripts/**`, `.github/workflows/**`

rollout 中の Phase 分割や review 経緯は archive を参照する。

- `docs/design/Go実装計画.md`
- `docs/design/archive/Go実装計画.rev34.md`

repo-backed policy への移行論点と branch/worktree 戦略は別紙を参照する。

- `docs/design/ポリシー永続化とブランチ戦略.md`

## 目的

merged PR を 1 件ずつ再実行し、現行 best ルールでの結果と、候補ルールを加えた結果を比較して、
改善したルールだけを昇格させる。学習はバッチではなく **1 PR = 1 run** の比較サイクルで進める。

## 設計原則

1. **Artifact-driven**: step 間の受け渡しはすべて `internal/contracts` に定義された JSON / JSONL artifact。
2. **Run-scoped**: 1 merged PR の処理は 1 `run_id` に閉じ込める。`run_id` 形式は `YYYY-MM-DD-PR<num>-<hex7>`。
3. **Strict contracts**: 読み書きは strict JSON、unknown field / duplicate key / trailing token を許さない。
4. **Two-pass comparison**: pass1 を baseline、pass2 を candidate-rules 適用版として比較する。
5. **Crash recovery first**: `processed.jsonl` と `intention.json` を使って途中停止から resume / recover できることを前提にする。

## 永続データモデル

1 run の durable artifact は `<runs_base>/<run_id>/` に置く。全 run 共通の正本は run ディレクトリの外に置く。

```text
<runs_base>/
├── <run_id>/
│   ├── task-package.json
│   ├── 20-pass1/a{1,2,3}/...
│   ├── 30/scores-A.jsonl
│   ├── 30/scores-A-raw.jsonl
│   ├── 30/compliance-A.jsonl
│   ├── 30/compliance-A-raw.jsonl
│   ├── 30/done.marker
│   ├── 40/candidates.json
│   ├── 40/classification.jsonl
│   ├── 50-pass2/a{1,2,3}/...
│   ├── 60/scores-B.jsonl
│   ├── 60/scores-B-raw.jsonl
│   ├── 60/compliance-B.jsonl
│   ├── 60/compliance-B-raw.jsonl
│   ├── 60/pairwise.jsonl
│   ├── 60/done.marker
│   └── 70/decision.json (+ adopt/rollback/needs_manual_recovery 中は intention.json)
├── processed.jsonl
├── needs-recovery/<run_id>.json
├── rules-registry.jsonl
├── rules-idempotency-index.jsonl
└── rules/<rule_id>.md
```

worktree は `runs/` の外に置く。step10 が **6 worktrees** を確保し、`task-package.json.worktrees[]` を正本として渡す。

- pass1: `a1`, `a2`, `a3`
- pass2: `a1`, `a2`, `a3`

## 主サイクル: 1 PR 完結の前後比較

### Step10: restore-base

- 対象 PR の merge-base と task 文脈を復元する
- `task-package.json` を書く
- pass1/pass2 用に 6 worktrees を確保する

### Step20: pass1 implement

- 現行 best ルールで 3 agents を並列実行する
- 各 agent は `manifest.json`, `session.jsonl`, `diff.patch`, `checklist-result.json` を書く
- checklist の自己申告 verdict は `[x]/[-]/[?]` ではなく **`compliant | n_a | exception`** を使う

### Step30: pass1 score

- pass1 成果物を judge panel で採点する
- 出力は final layer と raw layer を分ける
  - `<run_id>/30/scores-A.jsonl`
  - `<run_id>/30/scores-A-raw.jsonl`
  - `<run_id>/30/compliance-A.jsonl`
  - `<run_id>/30/compliance-A-raw.jsonl`

### Step40: extract-rules

- pass1 の採点理由、compliance 結果、実 PR との差分を材料に候補ルールを抽出する
- 旧 `candidate-rules.md` ではなく以下を出す
  - `40/candidates.json`
  - `40/classification.jsonl`

candidate は `new | update | duplicate` の 3 種で管理する。

### Step50: pass2 implement

- pass2 用の 3 worktrees を使い、best + candidate rules で再実装する
- 生成 artifact は step20 と同型だが `50-pass2/` 配下に出る

### Step60: pass2 score + pairwise

- pass2 も step30 と同じ rubric で採点する
- 出力は run-scoped file に固定する
  - `<run_id>/60/scores-B.jsonl`
  - `<run_id>/60/scores-B-raw.jsonl`
  - `<run_id>/60/compliance-B.jsonl`
  - `<run_id>/60/compliance-B-raw.jsonl`
  - `<run_id>/60/pairwise.jsonl`

pairwise 比較は **same-agent only** で行う。比較対象は `A_i vs B_i` であり、agent を跨いだ pairwise は作らない。

### Step70: decide + apply

- 判定結果は `decision.json` に `adopt | reject | noop | rollback` として書く
- adopt / rollback / needs_manual_recovery 中は `intention.json` を staged transaction の進捗記録として保持する
- best branch 更新と registry append は idempotent recovery 前提で進める

旧 `rules-next.md` は使わない。採用判断の durable artifact は `decision.json` と registry entry で表現する。

## 採点モデル

採点軸は旧設計の `Correctness/Design/Idiomatic/Fidelity/Discipline` ではなく、現在は次の 5 軸で固定する。

- `fidelity`
- `correctness`
- `maintainability`
- `discipline`
- `communication`

raw layer には judge role ごとの一次出力を残し、final layer には panel 解決後の確定 verdict を残す。
raw file は retention mandatory であり、後から削除しない。

## state と resume / recover

run の進行管理は `state.json` ではなく **`<runs_base>/processed.jsonl`** が正本である。

- non-terminal events: `started`, `step_done`, `interrupted`, `promoting`, `registry_size_high`, `registry_size_critical`, `rescue_retry`
- terminal events: `completed`, `failed`, `promoted`, `rollback`, `skipped`, `timeout`, `needs_manual_recovery`

step70 の adopt / rollback / needs_manual_recovery は `70/intention.json` で stage を管理し、`promotion.lock` を共有して recover / sunset と排他する。

## ルール registry と sunset

rules の durable lifecycle は `rules-registry.jsonl` で管理する。status は旧 `at-risk / removed` ではなく、
以下の 3 状態に整理する。

- `active`
- `deprecated`
- `archived`

status 遷移は registry entry kind で表現する。

- `status_changed`: `active -> deprecated` または `deprecated -> active`
- `archived`: `active | deprecated -> archived`
- `restored`: `archived -> active | deprecated`

rule 本体の長文は registry row に埋めず、`rules/<rule_id>.md` sidecar を参照する。

## 旧版からの差し替え点

この rewrite で、rev6 系の旧説明から以下を明示的に置き換える。

- `state.json` / `last_processed_pr` 中心の説明は廃止し、`processed.jsonl` ベースに更新した
- worktree 数は 3 ではなく **6** (`pass1 + pass2`, 各 3 agents)
- ルール候補・判定 artifact は `candidate-rules.md` / `rules-next.md` ではなく `candidates.json` / `classification.jsonl` / `decision.json` / `intention.json`
- 採点 artifact は run-scoped file name に固定した (`<run_id>/30/scores-A.jsonl` など)
- rubric 軸は `fidelity | correctness | maintainability | discipline | communication`
- checklist の自己申告は記号ではなく `compliant | n_a | exception`
- lifecycle は `active | deprecated | archived`

## まとめ

現在の pipeline は、1 PR を 2 pass で比較し、strict contract と append-only artifact を使って、
best ルールの昇格と rollback/recovery を安全に行う構成になっている。運用上の contract 正本は
`internal/contracts/**` と `docs/design/io-contracts.md` であり、この文書はその上位の構造説明を担当する。
