# ポリシー永続化とブランチ戦略

この文書は、現在の Go 実装における durable state の持ち方と、
repo-backed policy への移行方針を整理するための設計メモである。

field/schema の厳密な正本は `docs/design/io-contracts.md`、現行の全体像は
`docs/design/全体設計.md` を参照する。

## 現状

現在の durable state は 2 系統に分かれている。

1. `repo.best_branch`
   - step70 が promote 先として使う remote branch
   - winning pass2 commit (`target_sha`) の push / rollback / recover の対象

2. local `runsBase`
   - `rules-registry.jsonl`
   - `rules/*.md`
   - `processed.jsonl`
   - `needs-recovery/*.json`
   - run artifact

実際に judge / classify / pass2 prompt が参照する learned policy は
`runsBase/rules-registry.jsonl` と `runsBase/rules/*.md` であり、
`best_branch` ではない。

## 問題

この構成には次の欠点がある。

1. learned policy が machine-local になり、repo と一緒に運べない
2. `paths.runs` を変えると effective best policy が変わる
3. `best_branch` が更新されても、次回 run がそこを rule 正本として使わない
4. `best_branch` の意味が「winning code branch」と「best policy branch」で曖昧になる

## 目標

learned policy の durable 正本を対象 repo に置き、次回 run は必ずそこから始める。

## 段階的な理想像

### Phase A: repo-backed rules

まずは rules 系のみを repo に持つ。

- 新設: `repo.policy_branch`
- branch 上の managed paths:
  - `auto-improve/rules-registry.jsonl`
  - `auto-improve/rules/*.md`

local `runsBase` は runtime cache と journal として残すが、
rule 正本は `policy_branch` に一本化する。

### Phase B: broader policy overlay

将来的には rules だけでなく、managed path を拡張できる形にする。

候補:

- `CLAUDE.md`
- `AGENTS.md`
- Codex / agent 用設定
- AI rule files
- 必要なら opt-in の lint 設定

この段階では「ルール branch」ではなく「policy branch」として扱う方が自然である。

## ブランチ戦略

### `default_branch`

対象 repo の通常開発 branch。
merged PR の merge-base 復元元であり、step10 の immutable base を決める対象。

### `best_branch`

現行実装では winning pass2 commit の push 先。
この branch は code promotion / recover の remote pointer として扱う。

### `policy_branch`

learned policy の正本。
次回 run の harness/policy はここから hydrate する。

## worktree 戦略

### code worktrees

step10 が merged PR の merge-base から切る 6 本。

- pass1: `a1`, `a2`, `a3`
- pass2: `a1`, `a2`, `a3`

### policy snapshot

step10 は `policy_branch` の managed paths を local `runsBase` に hydrate する。
各 agent には local `runsBase` を通して同じ policy snapshot を見せる。

### adopt 時

step70 は local `runsBase` 上で確定した policy snapshot を `policy_branch` に publish する。

## pass1 / pass2 の意味

pass1 は `policy_branch@P0` を hydrate した状態で走る。

pass2 は pass1 の実装結果を持ち込むのではなく、
`P0 + candidate delta` を載せた fresh worktree で走る。

つまり:

- pass1: current policy
- pass2: current policy + candidate policy delta

## 勝敗後の整理

### adopt

- code 側: `best_branch` を promote
- policy 側: `policy_branch` を promote
- local worktree / heartbeat / rescue state は cleanup

### reject / noop

- `policy_branch` は動かさない
- local runtime artifact だけ残す

## 今回のバグ修理スコープ

まずは Phase A を対象にする。

1. `repo.policy_branch` を追加
2. `step10(HarnessFiles=true)` で `policy_branch` から local rules を hydrate
3. `step70 adopt` で local rules snapshot を `policy_branch` に publish
4. `paths.runs` が変わっても、repo + `policy_branch` が同じなら best policy は維持される

この修理では `best_branch` の code promotion semantics は変えない。
