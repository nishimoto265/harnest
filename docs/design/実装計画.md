## 方針サマリ

- 場所: **`auto-improve/`** (repo直下、既存 `scripts/` `docs/harness-eval/` とは独立、独立pnpm package)
- 言語: **TypeScript (Node.js 24, pnpm, tsx + zx)**
- 分解軸: **ステップ軸 + 薄い共通ライブラリ** (過剰抽象化回避)
- I/O契約: ファイル経由(jsonl中心、人向けはmd、設定はyaml)
- state管理: **append-only** (`processed.jsonl`)
- 自動化レベル: **全自動** (採用判定・クリーンアップ含む、人間介入なし)
- 実行モデル: **PR間直列(queue)、agent間並列(3体)**

- 将来的に Go 化する場合も JSON API 契約は同じ

## I/O 契約(各step)

### 配置規約

- すべて `runs/<date>-PR<num>/` 配下
- step番号を接頭辞に(例: `20-pass1/`, `30-scores-A.jsonl`)
- 成果物の絶対パス情報は `step-<N>.output.json` に集約して次stepへ渡す

### 主要ファイルのスキーマ

## 採用判定ロジック(step 70)

シンプル版:

```
win_count = Σ (pairwise.winner == "B")     # パス2がN体勝った

if win_count >= thresholds.adopt_win_count
   AND avg(scores_B.total) - avg(scores_A.total) >= adopt_score_delta_min:
  採用
  → rules_source に追加(new) or 置換(update)
  → best_branch に commit
  → candidates.md を archives/rejected/ には入れない(採用)
else:
  棄却
  → candidates.md を archives/rejected/YYYY-MM-DD_PR<num>.md に保存
  → best は変更しない

cleanup(worktree) # 採用/棄却問わず
state.append(processed.jsonl)
```

## 全自動化ポイント

| 要素 | 自動化方式 |
|---|---|
| PR検知 | `detect-merged.sh` をcronで定期実行(初期は手動起動も可) |

## Codexレビュー結果(3体並列)と対応案

2026-04-17にCodex 3体並列で adversarial-review を実施。
3体合計の指摘(Critical 3 / High 7 / Medium 4)を以下に反映する。

### 🔴 CRITICAL — 設計根幹の修正

#### C1. Same-PR overfitting(指摘2+3)→ サンセットサイクルで吸収

**問題(Codexの指摘)**: 1PR完結サイクルで抽出したルールを同PRで検証して即best昇格すると、そのPR特有の著者習慣・構造・クセを canonicalize してしまう。

**対応方針**:
- **trial段階は導入しない**(過剰複雑化を避ける)
- PR並列処理 + 後続PRで自然に holdout 検証される:
  - PR#74で採用されたルールは best に入る
  - PR#75以降のパス1で自動適用される → 悪影響があれば Discipline/compliance で可視化
  - サンセットサイクル(サイクル③)で閾値割れたら at-risk → removed
- 即時採用の"見落とし"は**サンセット機構が安全網**になる前提

**現状設計との整合**: 既存の「サンセット閾値でat-risk化」「3回連続at-riskで削除」のロジックが、この holdout 検証を兼ねる。**追加対応は不要**。

#### C2. Transactional promotion(指摘2)

**問題**: 現状の step 70 は best_branch commit → processed.jsonl 追記 の順序。間で落ちると branch は変更済み + 未処理扱いに。再実行で二重適用・競合。worktree自動削除で復旧不能。

**対応**: **State machine + lockでトランザクション化**。

```
採用決定時のstate遷移:
  proposed → promoting → promoted
                        ↓(失敗)
                        rollback
```

実装:
- 全state遷移を `processed.jsonl` に即座にappend(ロード時に最新stateを読む)
- best_branch commit の前に flock 取得
- commit成功後に promoted マーク、失敗時に rollback マーク+自動revert

#### C3. FIFO stale baseline(指摘1+2+3、全員一致)→ PR間完全直列化で解消

**問題**: 並列PR処理でeach PRが開始時のbestスナップショットで比較。FIFOで採用合流するが、先行PRの採用でbestが変わっても後続PRは古いbestで比較されたまま → 採用判定が測定時のbestに対して有効だが、合流後のbestに対しては無意味化。

**対応**: **PR間を完全直列化**する(queue処理)。

```
旧案(並列 + step 70シリアライズ):
  PR#74, #75, #76 を同時スタート、step 70だけ直列
  → 古いbest相手の検証 + 重複候補を捨てる無駄

新案(PR間完全直列):
  PR#74 完走 → PR#75 完走 → PR#76 完走
  → 常に最新bestで検証、重複検証の無駄ゼロ
```

**1PR内の agent 3体並列は維持**(pass1/pass2でエージェントを3体並列起動)。
PR間だけ直列化。

**実プロジェクト規模での見積もり**:
- 1PRあたり ≈ 1時間(agent並列込み)
- 週5PRなら累計5時間 → 夜間バッチで十分
- 個人・小規模運用では速度ペナルティ無視できる

**消える問題**:
- FIFO stale(常に最新bestで検証)
- 重複候補のskip問題(並列でない=重複そのものが発生しない)
- スナップショット管理/flock/compare-and-swap(全部不要)

---

### 🟠 HIGH — 仕様追加

#### H1. Task package契約(指摘1)

**問題**: step 10 入力が PR番号のみで、元タスクの指示(目的・対象ファイル・コンテキスト)を復元できない。

**対応**: step 10 出力に `task-package.json` 追加。

```json
{
  "pr_number": 74,
  "title": "...",
  "body": "...",
  "linked_issues": [...],
  "target_files_hint": [...],
  "acceptance_hints": [...],
  "reconstructed_task_prompt": "..."
}
```

`reconstructed_task_prompt` は PR本文+linked issues から LLM で生成(cheap_judgeモデル)。

#### H2. Rule registry構造化(指摘1)

**問題**: candidates.md + decision.json だけでは lifecycle metadata(status/history/contribution/compliance trend)を保持できない。

**対応**: `auto-improve/rules-registry.jsonl` を新設。1ルール1行、appendのみ(履歴保持)。

```jsonl
{"id":"AR-001","kind":"new","text_hash":"sha256:...","status":"trial","source_pr":74,"adopted_at":"...","check_method":"agent","metrics":{"fire_count":3,"contribution_avg":0.4,"compliance_rate":0.8}}
```

step 70/60/サンセットサイクルが共通参照する正本として扱う。

#### H3. Self-report complianceを決定パスから外す(指摘2)

**問題**: エージェント自己申告の checklist をDiscipline採点 → 採用/サンセットに反映している。self-reportはバイアスあり、Disciplineは既にworktree汚染で既知ノイズ。

**対応**:
- 決定パス(promotion/sunset)では static detector(grep/diff)優先
- self-reportは observation のみ(後で精度検証用)
- Disciplineは harness-injected files を除外して算出

#### H4. Provider budget + rate limit対策(指摘3)

**問題**: parallel_prs=3 × agents=3 × (Claude+Codex+arbiter) × 複数phase = 大量CLI呼び出し。既存rule-genで既にrate limit実害(A:8/13, B:10/13)。

**対応**: config.yaml に `provider_budget` を追加。

```yaml
providers:
  claude:
    max_concurrent: 5
    tokens_per_minute: 100000
    fallback: "single-judge"
  codex:
    max_concurrent: 3
    fallback: "skip"
```

rate limit検出時の自動fallback: panel → single-claude。

#### H5. Atomic completion manifest(指摘3)

**問題**: 既存harness-run.shでtimeout時に partial files が `done` 扱い。compliance.jsonl・pairwiseの信頼性崩壊。

**対応**: 各agent実行の完了定義を厳格化。

- agent完了 = `manifest.json` に session_jsonl hash, diff hash, checklist hash, exit code がcommit されたとき
- 採点ステップは manifest が無いagentを除外
- timeout時は kill → shutdown待ち → partial outputsを `quarantine/` へ移動、採点対象外

#### H6. Prompt injection対策(指摘2)

**問題**: `reason`/`evidence`/`problem`/`rationale` 等が自由text。下流LLMに instruction-shapedな文字列が流入すると誘導可能。

**対応**:
- agent生成物は strict JSON schema必須(free-textは許可するが separate field)
- 下流プロンプトへの埋め込み時に instruction-like patterns(例: `SYSTEM:`, `###`, code fenceなど)をescape
- 可能な限り non-LLM validation(diff一致、静的check)で代替

#### H7. Panel仲裁の中立化(指摘2)

**問題**: Claude判定 + Codex判定 → **メインClaudeが仲裁** = 片方の家族に寄る。

**対応**:
- Arbiter rotation: 仲裁者を iteration毎 or PR毎にClaude/Codex/(将来Gemini)でローテ
- Or deterministic arbitration: 理由の根拠(引用ファイル/行)の具体性で機械判定
- 当面は Codex arbiter を優先(パネル構成: Claude判定 + Codex判定 → Codex仲裁)

### 🟡 MEDIUM — 品質向上

#### M1. Checklist記号統一 + JSON併用(指摘1+2)

- 記号セットを `[x]` `[-]` `[?]` の**3種で統一**(`[!]`は廃止 = 違反自己申告は不要)
- `[ ]`(未記入)は集計時に別トラック化
- step 20で checklist.md と同時に `checklist-result.json` を emit

#### M2. processed.jsonl schema拡張(指摘1)

```jsonl
{"pr":74, "event":"started",   "at":"...", "run_id":"..."}
{"pr":74, "event":"step_done", "step":10, "at":"...", "run_id":"..."}
...
{"pr":74, "event":"promoting", "at":"...", "run_id":"..."}
{"pr":74, "event":"promoted",  "at":"...", "run_id":"..."}
```

event enum: `started / step_done / promoting / promoted / rollback / failed / timeout / skipped / completed`。
step の区別は `event="step_done"` + `step` field(10/20/30/40/50/60/70)で表現する。
(文字列enumに `step_10_done..step_70_done` を並べる案もあったが、集計容易性・拡張性の観点から `(event, step)` ペアに統一。)
冪等再開はrun_id + 最新stateで判定。

**TERMINAL set** (idempotency): `promoted / rollback / skipped / failed / completed / timeout`。
timeoutも terminal 扱いとし、自動リトライはしない(operatorが状況確認して手動で再キュー)。

#### M3. Worktree保持期間(指摘2)

cleanup タイミングを **probation(trial期間)完了まで** に延長。trial rule が production昇格 or 棄却されたタイミングで初めて worktree削除。

#### M4. Preflight hard gate + Python/Go検討(指摘3)

- preflight は OK/NG 判定ゲート化(NG時は detect-merged 実行禁止)
- 環境snapshotを run開始時に `runs/<dir>/env.json` に保存
- shell scriptの責務が膨らんだ段階で Python/Go化の再評価(M10 で判断)
